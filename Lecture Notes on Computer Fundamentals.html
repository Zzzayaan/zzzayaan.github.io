An array is a fixed sequence of indexed elements.
Arrays elements are accessed via their indices in time. E.g:

int[] a = { 1, 1, 2, 3, 5, 8 };
a[2]; // is equal to 2

In Java arrays are objects so therefore they're allocated dynamically using new, are manipulated by reference and are automatically deallocated when no longer accessible.

Array Equality
...

int[] x = new int[]{5,4,3,2,1};
int[] y = new int[]{5,4,3,2,1};

boolean test1 = x == y; // false
boolean test2 = x.equals(y); // false
boolean test3 = Arrays.equals(x, y); // true

The Arrays.equals() method returns true if if the two specified arrays of objects are equal to one another. Two arrays are equal if they contain the same elements in the same order. Also, two array references are considered equal if both are null.

2D Arrays
...

2D arrays are arrays with rows and columns:

double[][] a = new double[M][N]; // an MxN array

We test 2D array equality using the deepEquals() method:

int[][] x = new int[][]{{1}, {2, 3}};
int[][] y = new int[][]{{1}, {2, 3}};

boolean test1 = Arrays.equals(x, y); // false
boolean test2 = Arrays.deepEquals(x, y); // true

Lecture 2 - Test Driven Development
...

A traditional development method looks something like the following:
Analysis → Specification → Design → Implementation → Testing → Operation and maintenance

The main problem with this form of development is that it's hard to work to test, so problems are not always detected.

Test Driven Development is when you identify a (small) feature to be implemented, write and run a unit test for it, write smallest bit of code that will pass the test and run the test, repeating this process. The approach is to make small improvements frequently so that you always have a working system.

Unit tests should be focused, fast to run and independent of the environment and run order. An example of some tests for a method that sorts an array of integers:

  • Array is null
  • Array has zero length
  • Array has one element
  • All elements are the same
  • Array is already sorted
  • Array is reverse sorted etc.

JUnit
...

JUnit is a testing framework in Java which facilitates the running and writing of tests. It can declare methods as tests and setups, and has its own methods to check conditions, such as assertEquals and assertThrows.

// an example of some JUnit tests
public class AccountTests {
	private Account a;
	
	@BeforeEach
	public void setupAccount() {
		a = new Account();
	}
	@Test
	public void withdrawZero() {
		a.withdraw(0);
		assertEquals(0, a.getBalance());
	}
	@Test
	public void withdrawTen() {
		a.withdraw(10);
		assertEquals(-10, a.getBalance());
	}
}

Lecture 3 - Collections (ADT interfaces)
...

We're going to define the ADT interface 'collection' using Java primitives, and we'll also run unit tests to understand expected behaviour. Our implementation of the collection will use arrays as the underlying data structure.

For our collection interface, we'd like to do the following:

  • Add things
  • Remove things
  • Check if a given thing is in the collection
  • Check how big the collection is
  • Ask what's at a given position in the collection
public interface ICollection {
	void add(Object obj);
	void remove(Object obj);
	void removeAll(Object obj);
	boolean contains(Object obj);
	int size();
	Object elementAt(int index);
}

We should also write our test cases, such as:

  • Add an element and check size
  • Add and remove an element
  • Add two elements and remove one etc.
    These tests are sketches as to how we expect the ADT to behave.

Below are examples of some test cases written in Java, using the JUnit framework:

@Test
public void addAndRemoveOneElement() {
	int element = 3;
	collection.add(element);
	collection.remove(element);
	assertFalse(collection.contains(element));
	assertEquals(0, collection.size());
}

@Test
public void addTooMany() {
	collection = new ArrayCollection(2);
	collection.add(5);
	collection.add(6);
	assertThrows(IllegalStateException.class, () -> {
		collection.add(7);
	});
}

@Test
public void getElementOutOfBounds() {
	collection.add(3);
	assertThrows(IllegalArgumentException.class, () -> {
	collection.elementAt(1);
	});
}

Here's an example of how we can use our collections; in this case we want to create a collection of bank accounts:

ICollection accounts = new ArrayCollection();

accounts.add(new Account("jane", 240));
accounts.add(new Account("john", 0));
accounts.add(new Account("jill", -3000));
accounts.add(new Account("janet", 7));

int total = 0;
for (int i = 0; i < accounts.size(); i++) {
	Account account = (Account)accounts.elementAt(i);
	total += account.getBalance();
}

Finally, we just have to do the implementation of the ArrayCollection() class (keep in mind this is only a partial implementation):

public class ArrayCollection implements ICollection {
	private Object[] store; // array for storing the objects
	private int current_size = 0; // the number of objects currently stored.
	private static final int DEFAULT_CAPACITY = 10; // the default capacity of the storage array.
	
	public ArrayCollection(int capacity) {
		store = new Object[capacity];
	}
	public ArrayCollection() {
		this(DEFAULT_CAPACITY);
	}
	public void add(Object obj) {
	if (current_size < store.length)
	store[current_size++] = obj;
	else throw new IllegalStateException();
	}
...}

Since the underlying primitive type for this ADT is an array, the time complexities are as the follows:

  • add, size, elementAt → constant
  • contains, remove → linear

Lecture 4 - Factories
...

Following on from previous lectures, we saw that we could implement interfaces into classes, to then run unit tests on to check if it works expectedly. However, this is how it would currently look like:

public class ArrayCollectionTest1 {
	private ArrayCollection collection =
	new ArrayCollection();
	...
	// 100s of test methods
}

public class ArrayCollectionTest2 {
	private ICollection collection =
	new ArrayCollection();
	...
	// 100s of test methods
}
// and so forth...

Instead of this tedious and repetitive code, we can use factories which abstract over concrete classes used in implementations. The idea is to create a factory class that is only used where concrete implementation needs to be referred to.

Factories provide an interface for creating objects, allowing subclasses to alter the type of objects that will be created.

There are variations in how you can implement the factory class. The most common are:

  1. Defining separate factory classes for each concrete class:
interface ShapeFactory {
    Shape createShape();
}

class CircleFactory implements ShapeFactory {
    @Override
    public Shape createShape() {
        return new Circle();
    }
}

class RectangleFactory implements ShapeFactory {
    @Override
    public Shape createShape() {
        return new Rectangle();
    }
}
  1. Passing an instance of factory to code that needs to create things:
class Client {
    private final ShapeFactory factory;

    public Client(ShapeFactory factory) {
        this.factory = factory;
    }

    public void createShape() {
        Shape shape = factory.createShape();
        // Use the created shape
    }
}
  1. Single factory class that decides which concrete class to instantiate:
class ShapeFactory {
    public Shape createShape(String shapeType) {
        if (shapeType.equalsIgnoreCase("Circle")) {
            return new Circle();
        } else if (shapeType.equalsIgnoreCase("Rectangle")) {
            return new Rectangle();
        }
        return null;
    }
}

Searching for Element
...

Searching for an element can be done more efficiently than a linear scan if we know the array is already sorted.
For example, we could perform a binary search which works as the follows:

  • Check the middle element
  • Greater than, equal to, or less than target...
  • If not found, search again in lower or upper half
  • Repeat until target is found

An iterative binary search in Java:

private int binarySplit(Comparable element) {
	int lower = 0; // lower bound of the array
	int upper = current_size - 1; // upper bound of the array
	
	while (lower <= upper) {
		int middle = (lower + upper) / 2;
		int compare = store[middle].compareTo(element);
		if (compare == 0) return middle;
		if (compare < 0)
		lower = middle + 1;
		else
		upper = middle - 1;
	}
	return lower;
}

A recursive implementation:

private int recBinarySplit(Comparable element) {
	return recBinarySplit(element, 0, current_size - 1);
}

private int recBinarySplit(Comparable element, int lower, int upper) {
	if (lower > upper) return lower;
	int middle = (lower + upper) / 2;
	int compare = store[middle].compareTo(element);
	if (compare == 0) return middle;
	if (compare < 0)
	return recBinarySplit(element, middle + 1, upper);
	return recBinarySplit(element, lower, middle - 1);
}

Now we can implement some of the common methods applied on arrays with this implementation, such as:

  1. contains()
public boolean contains(Comparable element) {
	int index = binarySplit(element);
	return index != current_size &&
	store[index].equals(element);
}
  1. remove()
public void remove(Comparable element) {
	int index = binarySplit(element);
	if (index != current_size &&
	store[index].equals(element)) {
		for (; index < current_size - 1; index++)
		store[index] = store[index + 1];
		current_size--;
	}
}
  1. add()
public void add(Comparable element) {
	if (current_size < store.length) {
		int index = binarySplit(element);
		if (index == current_size)
		store[current_size++] = element;
		else {
			for (int i = current_size; i > index; i--)
			store[i] = store[i - 1];
			store[index] = element;
			current_size++;
		}
	}
	else throw new IllegalStateException();
}

Now, if we look at the complexity of the same operations on sorted arrays, we find:

  • add, remove → linear
  • size, elementAt → constant
  • contains → logarithmic
    In the end, the tradeoff we get here is that a sorted array make searching more efficient and insertion less efficient.
